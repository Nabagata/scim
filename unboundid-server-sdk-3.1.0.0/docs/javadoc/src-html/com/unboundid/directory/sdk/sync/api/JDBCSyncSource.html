060    /**
061     * This class defines an API that must be implemented by extensions
062     * in order to synchronize data out of a relational database. Since the
063     * UnboundID Synchronization Server is LDAP-centric,
064     * this API allows you to take database content and convert it into LDAP
065     * entries which can then be processed by the Synchronization Server. The
066     * lifecycle of
067     * a sync operation is as follows:
068     * <ol>
069     * <li>Detect change at the synchronization source</li>
070     * <li>Fetch full source entry</li>
071     * <li>Perform any mappings and compute the equivalent destination entry</li>
072     * <li>Fetch full destination entry</li>
073     * <li>Diff the computed destination entry and actual (fetched) destination
074     * entry</li>
075     * <li>Apply the minimal set of changes at the destination to bring it in sync
076     * </li>
077     * </ol>
078     * This implies that the
079     * {@link #fetchEntry(TransactionContext, SyncOperation)} method will be
080     * called once for every change that is returned by
081     * {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)}.
082     * <p>
083     * In several places a {@link TransactionContext} is provided, which allows
084     * controlled access to the target database. By default, methods in this class
085     * are always provided with a fresh connection (i.e. a new transaction), and the
086     * Synchronization Server will always commit or rollback the transaction
087     * automatically, depending on whether the method returned normally or threw an
088     * exception. Implementers may optionally perform their own transaction
089     * management within these methods if necessary. The TransactionContext also
090     * provides logging facilities and allows a timeout, which will automatically
091     * invalidate the context object after the specified duration.
092     * <p>
093     * Several of these methods throw {@link SQLException}, which should be used in
094     * the case of any database access error. For other types of errors, runtime
095     * exceptions may be used (IllegalStateException, NullPointerException, etc.).
096     * The Synchronization Server will automatically retry operations that fail, up
097     * to a configurable amount of attempts. The exception to this rule is if a
098     * SQLException is thrown with a SQL state string beginning with "08"; this
099     * indicates a connection error, and in this case the operation is retried
100     * indefinitely.
101     * <p>
102     * Implementers need not worry about the cost of repeatedly parsing and
103     * preparing SQL statements; all statements will be automatically cached.
104     * <BR>
105     * <H2>Configuring JDBC Sync Sources</H2>
106     * In order to configure a JDBC sync source based on this API and
107     * written in Java, use a command like:
108     * <PRE>
109     *      dsconfig create-sync-source \
110     *           --source-name "<I>{source-name}</I>" \
111     *           --type third-party-jdbc \
112     *           --set "server:{server-name}" \
113     *           --set "extension-class:<I>{class-name}</I>" \
114     *           --set "extension-argument:<I>{name=value}</I>"
115     * </PRE>
116     * where "<I>{source-name}</I>" is the name to use for the JDBC sync source
117     * instance, "<I>{server-name}</I>" is the name of the JDBC external server that
118     * will be used as the sync source, "<I>{class-name}</I>" is the fully-qualified
119     * name of the Java class written using this API, and "<I>{name=value}</I>"
120     * represents name-value pairs for any arguments to provide to the JDBC sync
121     * source.  If multiple arguments should be provided to the JDBC sync source,
122     * then the "<CODE>--set extension-argument:<I>{name=value}</I></CODE>" option
123     * should be provided multiple times.
124     */

232       * This method should effectively set the starting point for synchronization
233       * to the place specified by the <code>options</code> parameter. This should
234       * cause all changes previous to the specified start point to be disregarded
235       * and only changes after that point to be returned by
236       * {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)}.
237       * <p>
238       * There are several different startpoint types (see
239       * {@link SetStartpointOptions}), and this implementation is not required to
240       * support them all. If the specified startpoint type is unsupported, this
241       * method should throw an {@link IllegalArgumentException}.
242       * <p>
243       * <b>IMPORTANT</b>: The <code>RESUME_AT_SERIALIZABLE</code> startpoint type
244       * must be supported by your implementation, because this is used when a Sync
245       * Pipe first starts up.
246       * <p>
247       * This method can be called from two different contexts:
248       * <ul>
249       * <li>When the 'set-startpoint' subcommand of the realtime-sync CLI is used
250       * (the Sync Pipe is required to be stopped in this context)</li>
251       * <li>Immediately after a connection is first established to the source
252       * server (e.g. before the first call to
253       * {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)})</li>
254       * </ul>
255       * @param ctx
256       *          a TransactionContext which provides a valid JDBC connection to the
257       *          database.
258       * @param options
259       *          an object which indicates where exactly to start synchronizing
260       *          (e.g.
261       *          the end of the changelog, specific change number, a certain time
262       *          ago, etc)
263       * @throws SQLException
264       *           if there is any error while setting the start point
265       */

273      /**
274       * Gets the current value of the startpoint for change detection. This is the
275       * "bookmark" which indicates which changes have already been processed and
276       * which have not. In most cases, a change number is used to detect changes
277       * and is managed by the Synchronization Server, in which case this
278       * implementation needs only to return the latest acknowledged
279       * change number. In other cases, the return value may correspond to a
280       * different value, such as the SYS_CHANGE_VERSION in Microsoft SQL Server.
281       * In any case, this method should return the value that is updated by
282       * {@link #acknowledgeCompletedOps(TransactionContext, LinkedList)}.
283       * <p>
284       * This method is called periodically and the return value is saved in the
285       * persistent state for the Sync Pipe that uses this script as its Sync
286       * Source.
287       * <p>
288       * <b>IMPORTANT</b>: The internal value for the startpoint should only be
289       * updated after a sync operation is acknowledged back to this script (via
290       * {@link #acknowledgeCompletedOps(TransactionContext, LinkedList)}).
291       * Otherwise it will be possible for changes to be missed when the
292       * Synchronization Server is restarted or a connection error occurs.
293       * @return a value to store in the persistent state for the Sync Pipe. This is
294       *         usually
295       *         a change number, but if a changelog table is not used to detect
296       *         changes,
297       *         this value should represent some other token to pass into
298       *         {@link #setStartpoint(TransactionContext, SetStartpointOptions)}
299       *         when the sync pipe starts up.
300       */
301      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
302      public abstract Serializable getStartpoint();

306      /**
307       * Return the next batch of change records from the database. Change records
308       * are just hints that a change happened; they do not include the actual data
309       * of the change. In an effort to never synchronize stale data, the
310       * Synchronization Server will go back and fetch the full source entry for
311       * each change record.
312       * <p>
313       * On the first invocation, this should return changes starting from the
314       * startpoint that was set by
315       * {@link #setStartpoint(TransactionContext, SetStartpointOptions)}. This
316       * method is responsible for updating the internal state such that subsequent
317       * invocations do not return duplicate changes.
318       * <p>
319       * The resulting list should be limited by <code>maxChanges</code>. The
320       * <code>numStillPending</code> reference should be set to the estimated
321       * number of changes that haven't yet been retrieved from the changelog table
322       * when this method returns, or zero if all the current changes have been
323       * retrieved.
324       * <p>
325       * <b>IMPORTANT</b>: While this method needs to keep track of which changes
326       * have already been returned so that it does not return them again, it should
327       * <b>NOT</b> modify the official startpoint. The internal value for the
328       * startpoint should only be updated after a sync operation is acknowledged
329       * back to this script (via
330       * {@link #acknowledgeCompletedOps(TransactionContext, LinkedList)}).
331       * Otherwise it will be possible for changes to be missed when the
332       * Synchronization Server is restarted or a connection error occurs. The
333       * startpoint should not change as a result of this method.
334       * <p>
335       * This method <b>does not need to be thread-safe</b>. It will be invoked
336       * repeatedly by a single thread, based on the polling interval set in the
337       * Sync Pipe configuration.
338       * @param ctx
339       *          a TransactionContext which provides a valid JDBC connection to the
340       *          database.
341       * @param maxChanges
342       *          the maximum number of changes to retrieve
343       * @param numStillPending
344       *          this should be set to the number of unretrieved changes that
345       *          are still pending after this batch has been retrieved. This will
346       *          be passed in
347       *          as zero, and may be left that way if the actual value cannot be
348       *          determined.
349       * @return a list of {@link DatabaseChangeRecord} instances, each
350       *         corresponding
351       *         to a row in the changelog table (or the equivalent if some other
352       *         change
353       *         tracking mechanism is being used). If there are no new changes to
354       *         return, this
355       *         method should return an empty list.
356       * @throws SQLException
357       *           if there is any error while retrieving the next batch of changes
358       */
359      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
360      public abstract List<DatabaseChangeRecord> getNextBatchOfChanges(
361                                                  final TransactionContext ctx,
362                                                  final int maxChanges,
363                                                  final AtomicLong numStillPending)
364                                                          throws SQLException;
365

369       * Return a full source entry (in LDAP form) from the database, corresponding
370       * to the {@link DatabaseChangeRecord} that is passed in through the
371       * {@link SyncOperation}. This method should perform any queries necessary to
372       * gather the latest values for all the attributes to be synchronized.
373       * <p>
374       * This method <b>must be thread safe</b>, as it will be called repeatedly and

376       * entries.
377       * @param ctx
378       *          a TransactionContext which provides a valid JDBC connection to the
379       *          database.
380       * @param operation
381       *          the SyncOperation which identifies the database "entry" to
382       *          fetch. The DatabaseChangeRecord can be obtained by calling
383       *          <code>operation.getDatabaseChangeRecord()</code>.
384       *          This is returned by

386       *          or by
387       *        {@link #listAllEntries(TransactionContext, String, BlockingQueue)}
388       *          .
389       * @return a full LDAP Entry, or null if no such entry exists.
390       * @throws SQLException
391       *           if there is an error fetching the entry

400      /**
401       * Provides a way for the Synchronization Server to acknowledge back to the
402       * script which sync operations it has processed. This method should update
403       * the official startpoint which was set by
404       * {@link #setStartpoint(TransactionContext, SetStartpointOptions)} and is
405       * returned by {@link #getStartpoint()}.
406       * <p>
407       * <b>IMPORTANT</b>: The internal value for the startpoint should only be
408       * updated after a sync operation is acknowledged back to this script (via
409       * this method). Otherwise it will be possible for changes to be missed when
410       * the Synchronization Server is restarted or a connection error occurs.
411       * <p>
412       * A {@link TransactionContext} is provided in case the acknowledgment needs
413       * to make it all the way back to the database itself (for example if you were
414       * using Oracle's Change Data Capture).
415       * @param ctx
416       *          a TransactionContext which provides a valid JDBC connection to the
417       *          database.
418       * @param completedOps
419       *          a list of {@link SyncOperation}s that have finished processing.
420       *          The records are listed in the order they were first detected.
421       * @throws SQLException
422       *           if there is an error acknowledging the changes back to the
423       *           database
424       */
425      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
426      public abstract void acknowledgeCompletedOps(
427                                       final TransactionContext ctx,
428                                       final LinkedList<SyncOperation> completedOps)

468      /**
469       * Gets a list of all the entries in the database for a given entry type. This
470       * is used by the 'resync' command line tool. The default implementation
471       * throws a {@link UnsupportedOperationException}; subclasses should override
472       * if the resync functionality is needed.
473       * <p>
474       * The <code>entryType</code> is user-defined; it will be
475       * passed in on the command line for resync. The <code>outputQueue</code>
476       * should contain {@link DatabaseChangeRecord} objects with the
477       * <code>ChangeType</code> set to <i>resync</i>.
478       * <p>
479       * This method should not return until all the entries of the given entryType
480       * have been added to the output queue. Separate threads will concurrently
481       * drain entries from the queue and process them. The queue should not
482       * actually contain full entries, but rather DatabaseChangeRecord objects
483       * which identify the full database entries. These objects are then
484       * individually passed in to
485       * {@link #fetchEntry(TransactionContext, SyncOperation)}. Therefore,
486       * it is important to make sure that the DatabaseChangeRecord instances
487       * contain enough identifiable information (e.g. primary keys) for each entry
488       * so that the entry can be found again.
489       * <p>
490       * The lifecycle of resync is similar to that of real-time sync, with a few
491       * differences:
492       * <ol>
493       * <li>Stream out a list of all IDs in the database (for a given entryType)
494       * </li>
495       * <li>Fetch full source entry for an ID</li>
496       * <li>Perform any mappings and compute the equivalent destination entry</li>
497       * <li>Fetch full destination entry</li>
498       * <li>Diff the computed destination entry and actual destination entry</li>
499       * <li>Apply the minimal set of changes at the destination to bring it in sync
500       * </li>
501       * </ol>
502       * If the total set of entries is very large, it is fine to split up the work
503       * into multiple database queries within this method. The queue will not grow
504       * out of control because it blocks when it becomes full. The queue capacity
505       * is fixed at 1000.
506       * <p>
507       * @param ctx
508       *          a TransactionContext which provides a valid JDBC connection to the
509       *          database.
510       * @param entryType
511       *          the type of database entry to be fetched (this is specified
512       *          on the CLI for the resync command)
513       * @param outputQueue
514       *          a queue of DatabaseChangeRecord objects which will be individually
515       *          fetched via {@link #fetchEntry(TransactionContext, SyncOperation)}
516       * @throws SQLException
517       *           if there is an error retrieving the list of entries to resync
518       */
519      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
520      public void listAllEntries(final TransactionContext ctx,
521                                 final String entryType,
522                                 final BlockingQueue<DatabaseChangeRecord>
523                                           outputQueue) throws SQLException
524      {
525        throw new UnsupportedOperationException();
526      }
