052    /**
053     * This class defines an API that must be implemented by scripted extensions
054     * in order to synchronize data out of a relational database. Since the
055     * UnboundID Synchronization Server is LDAP-centric,
056     * this API allows you to take database content and convert it into LDAP
057     * entries which can then be processed by the Synchronization Server. The
058     * lifecycle of
059     * a sync operation is as follows:
060     * <ol>
061     * <li>Detect change at the synchronization source</li>
062     * <li>Fetch full source entry</li>
063     * <li>Perform any mappings and compute the equivalent destination entry</li>
064     * <li>Fetch full destination entry</li>
065     * <li>Diff the computed destination entry and actual (fetched) destination
066     * entry</li>
067     * <li>Apply the minimal set of changes at the destination to bring it in sync
068     * </li>
069     * </ol>
070     * This implies that the
071     * {@link #fetchEntry(TransactionContext, SyncOperation)} method will be
072     * called once for every change that is returned by
073     * {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)}.
074     * <p>
075     * In several places a {@link TransactionContext} is provided, which allows
076     * controlled access to the target database. By default, methods in this class
077     * are always provided with a fresh connection (i.e. a new transaction), and the
078     * Synchronization Server will always commit or rollback the transaction
079     * automatically, depending on whether the method returned normally or threw an
080     * exception. Implementers may optionally perform their own transaction
081     * management within these methods if necessary. The TransactionContext also
082     * provides logging facilities and allows a timeout, which will automatically
083     * invalidate the context object after the specified duration.
084     * <p>
085     * Several of these methods throw {@link SQLException}, which should be used in
086     * the case of any database access error. For other types of errors, runtime
087     * exceptions may be used (IllegalStateException, NullPointerException, etc.).
088     * The Synchronization Server will automatically retry operations that fail, up
089     * to a configurable amount of attempts. The exception to this rule is if a
090     * SQLException is thrown with a SQL state string beginning with "08"; this
091     * indicates a connection error, and in this case the operation is retried
092     * indefinitely.
093     * <p>
094     * Implementers need not worry about the cost of repeatedly parsing and
095     * preparing SQL statements; all statements will be automatically cached.
096     * <BR>
097     * <H2>Configuring Groovy-Scripted JDBC Sync Sources</H2>
098     * In order to configure a scripted JDBC sync source based on this API and
099     * written in the Groovy scripting language, use a command like:
100     * <PRE>
101     *      dsconfig create-sync-source \
102     *           --source-name "<I>{source-name}</I>" \
103     *           --type groovy-scripted-jdbc \
104     *           --set "server:{server-name}" \
105     *           --set "script-class:<I>{class-name}</I>" \
106     *           --set "script-argument:<I>{name=value}</I>"
107     * </PRE>
108     * where "<I>{source-name}</I>" is the name to use for the JDBC sync source
109     * instance, "<I>{server-name}</I>" is the name of the JDBC external server that
110     * will be used as the sync source, "<I>{class-name}</I>" is the fully-qualified
111     * name of the Groovy class written using this API, and "<I>{name=value}</I>"
112     * represents name-value pairs for any arguments to provide to the JDBC sync
113     * source.  If multiple arguments should be provided to the JDBC sync source,
114     * then the "<CODE>--set script-argument:<I>{name=value}</I></CODE>" option
115     * should be provided multiple times.
116     */

177      /**
178       * This method should effectively set the starting point for synchronization
179       * to the place specified by the <code>options</code> parameter. This should
180       * cause all changes previous to the specified start point to be disregarded
181       * and only changes after that point to be returned by
182       * {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)}.
183       * <p>
184       * There are several different startpoint types (see
185       * {@link SetStartpointOptions}), and this implementation is not required to
186       * support them all. If the specified startpoint type is unsupported, this
187       * method should throw an {@link IllegalArgumentException}.
188       * <p>
189       * <b>IMPORTANT</b>: The <code>RESUME_AT_SERIALIZABLE</code> startpoint type
190       * must be supported by your implementation, because this is used when a Sync
191       * Pipe first starts up.
192       * <p>
193       * This method can be called from two different contexts:
194       * <ul>
195       * <li>When the 'set-startpoint' subcommand of the realtime-sync CLI is used
196       * (the Sync Pipe is required to be stopped in this context)</li>
197       * <li>Immediately after a connection is first established to the source
198       * server (e.g. before the first call to
199       * {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)})</li>
200       * </ul>
201       * @param ctx
202       *          a TransactionContext which provides a valid JDBC connection to the
203       *          database.
204       * @param options
205       *          an object which indicates where exactly to start synchronizing
206       *          (e.g.
207       *          the end of the changelog, specific change number, a certain time
208       *          ago, etc)
209       * @throws SQLException
210       *           if there is any error while setting the start point
211       */
212      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
213      public abstract void setStartpoint(final TransactionContext ctx,
214                                         final SetStartpointOptions options)
215                                              throws SQLException;

217      /**
218       * Gets the current value of the startpoint for change detection. This is the
219       * "bookmark" which indicates which changes have already been processed and
220       * which have not. In most cases, a change number is used to detect changes
221       * and is managed by the Synchronization Server, in which case this
222       * implementation needs only to return the latest acknowledged
223       * change number. In other cases, the return value may correspond to a
224       * different value, such as the SYS_CHANGE_VERSION in Microsoft SQL Server.
225       * In any case, this method should return the value that is updated by
226       * {@link #acknowledgeCompletedOps(TransactionContext, LinkedList)}.
227       * <p>
228       * This method is called periodically and the return value is saved in the
229       * persistent state for the Sync Pipe that uses this script as its Sync
230       * Source.
231       * <p>
232       * <b>IMPORTANT</b>: The internal value for the startpoint should only be
233       * updated after a sync operation is acknowledged back to this script (via
234       * {@link #acknowledgeCompletedOps(TransactionContext, LinkedList)}).
235       * Otherwise it will be possible for changes to be missed when the
236       * Synchronization Server is restarted or a connection error occurs.
237       * @return a value to store in the persistent state for the Sync Pipe. This is
238       *         usually
239       *         a change number, but if a changelog table is not used to detect
240       *         changes,
241       *         this value should represent some other token to pass into
242       *         {@link #setStartpoint(TransactionContext, SetStartpointOptions)}
243       *         when the sync pipe starts up.
244       */
245      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
246      public abstract Serializable getStartpoint();
247    
248      /**
249       * Return the next batch of change records from the database. Change records
250       * are just hints that a change happened; they do not include the actual data
251       * of the change. In an effort to never synchronize stale data, the
252       * Synchronization Server will go back and fetch the full source entry for
253       * each change record.
254       * <p>
255       * On the first invocation, this should return changes starting from the
256       * startpoint that was set by
257       * {@link #setStartpoint(TransactionContext, SetStartpointOptions)}. This
258       * method is responsible for updating the internal state such that subsequent
259       * invocations do not return duplicate changes.
260       * <p>
261       * The resulting list should be limited by <code>maxChanges</code>. The
262       * <code>numStillPending</code> reference should be set to the estimated
263       * number of changes that haven't yet been retrieved from the changelog table
264       * when this method returns, or zero if all the current changes have been
265       * retrieved.
266       * <p>
267       * <b>IMPORTANT</b>: While this method needs to keep track of which changes
268       * have already been returned so that it does not return them again, it should
269       * <b>NOT</b> modify the official startpoint. The internal value for the
270       * startpoint should only be updated after a sync operation is acknowledged
271       * back to this script (via
272       * {@link #acknowledgeCompletedOps(TransactionContext, LinkedList)}).
273       * Otherwise it will be possible for changes to be missed when the
274       * Synchronization Server is restarted or a connection error occurs. The
275       * startpoint should not change as a result of this method.
276       * <p>
277       * This method <b>does not need to be thread-safe</b>. It will be invoked
278       * repeatedly by a single thread, based on the polling interval set in the
279       * Sync Pipe configuration.
280       * @param ctx
281       *          a TransactionContext which provides a valid JDBC connection to the
282       *          database.
283       * @param maxChanges
284       *          the maximum number of changes to retrieve
285       * @param numStillPending
286       *          this should be set to the number of unretrieved changes that
287       *          are still pending after this batch has been retrieved. This will
288       *          be passed in
289       *          as zero, and may be left that way if the actual value cannot be
290       *          determined.
291       * @return a list of {@link DatabaseChangeRecord} instances, each
292       *         corresponding
293       *         to a row in the changelog table (or the equivalent if some other
294       *         change
295       *         tracking mechanism is being used). If there are no new changes to
296       *         return, this
297       *         method should return an empty list.
298       * @throws SQLException
299       *           if there is any error while retrieving the next batch of changes
300       */
301      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
302      public abstract List<DatabaseChangeRecord> getNextBatchOfChanges(
303                                                  final TransactionContext ctx,
304                                                  final int maxChanges,
305                                                  final AtomicLong numStillPending)
306                                                          throws SQLException;
307    
308      /**
309       * Return a full source entry (in LDAP form) from the database, corresponding
310       * to the {@link DatabaseChangeRecord} that is passed in through the
311       * {@link SyncOperation}. This method should perform any queries necessary to
312       * gather the latest values for all the attributes to be synchronized.
313       * <p>
314       * This method <b>must be thread safe</b>, as it will be called repeatedly and
315       * concurrently by each of the Sync Pipe worker threads as they process
316       * entries.
317       * @param ctx
318       *          a TransactionContext which provides a valid JDBC connection to the
319       *          database.
320       * @param operation
321       *          the SyncOperation which identifies the database "entry" to
322       *          fetch. The DatabaseChangeRecord can be obtained by calling
323       *          <code>operation.getDatabaseChangeRecord()</code>.
324       *          This is returned by
325       *        {@link #getNextBatchOfChanges(TransactionContext, int, AtomicLong)}
326       *          or by
327       *        {@link #listAllEntries(TransactionContext, String, BlockingQueue)}
328       *          .
329       * @return a full LDAP Entry, or null if no such entry exists.
330       * @throws SQLException
331       *           if there is an error fetching the entry
332       */
333      @ThreadSafety(level = ThreadSafetyLevel.METHOD_THREADSAFE)
334      public abstract Entry fetchEntry(final TransactionContext ctx,
335                                       final SyncOperation operation)
336                                                throws SQLException;

338      /**
339       * Provides a way for the Synchronization Server to acknowledge back to the
340       * script which sync operations it has processed. This method should update
341       * the official startpoint which was set by
342       * {@link #setStartpoint(TransactionContext, SetStartpointOptions)} and is
343       * returned by {@link #getStartpoint()}.
344       * <p>
345       * <b>IMPORTANT</b>: The internal value for the startpoint should only be
346       * updated after a sync operation is acknowledged back to this script (via
347       * this method). Otherwise it will be possible for changes to be missed when
348       * the Synchronization Server is restarted or a connection error occurs.
349       * <p>
350       * A {@link TransactionContext} is provided in case the acknowledgment needs
351       * to make it all the way back to the database itself (for example if you were
352       * using Oracle's Change Data Capture).
353       * @param ctx
354       *          a TransactionContext which provides a valid JDBC connection to the
355       *          database.
356       * @param completedOps
357       *          a list of {@link SyncOperation}s that have finished processing.
358       *          The records are listed in the order they were
359       *          first detected.
360       * @throws SQLException
361       *           if there is an error acknowledging the changes back to the
362       *           database
363       */
364      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
365      public abstract void acknowledgeCompletedOps(
366                                       final TransactionContext ctx,
367                                       final LinkedList<SyncOperation> completedOps)
368                                                 throws SQLException;

403      /**
404       * Gets a list of all the entries in the database for a given entry type. This
405       * is used by the 'resync' command line tool. The default implementation
406       * throws a {@link UnsupportedOperationException}; subclasses should override
407       * if the resync functionality is needed.
408       * <p>
409       * The <code>entryType</code> is user-defined; it will be
410       * passed in on the command line for resync. The <code>outputQueue</code>
411       * should contain {@link DatabaseChangeRecord} objects with the
412       * <code>ChangeType</code> set to <i>resync</i>.
413       * <p>
414       * This method should not return until all the entries of the given entryType
415       * have been added to the output queue. Separate threads will concurrently
416       * drain entries from the queue and process them. The queue should not
417       * actually contain full entries, but rather DatabaseChangeRecord objects
418       * which identify the full database entries. These objects are then
419       * individually passed in to
420       * {@link #fetchEntry(TransactionContext, SyncOperation)}. Therefore,
421       * it is important to make sure that the DatabaseChangeRecord instances
422       * contain enough identifiable information (e.g. primary keys) for each entry
423       * so that the entry can be found again.
424       * <p>
425       * The lifecycle of resync is similar to that of real-time sync, with a few
426       * differences:
427       * <ol>
428       * <li>Stream out a list of all IDs in the database (for a given entryType)
429       * </li>
430       * <li>Fetch full source entry for an ID</li>
431       * <li>Perform any mappings and compute the equivalent destination entry</li>
432       * <li>Fetch full destination entry</li>
433       * <li>Diff the computed destination entry and actual destination entry</li>
434       * <li>Apply the minimal set of changes at the destination to bring it in sync
435       * </li>
436       * </ol>
437       * If the total set of entries is very large, it is fine to split up the work
438       * into multiple database queries within this method. The queue will not grow
439       * out of control because it blocks when it becomes full. The queue capacity
440       * is fixed at 1000.
441       * <p>
442       * @param ctx
443       *          a TransactionContext which provides a valid JDBC connection to the
444       *          database.
445       * @param entryType
446       *          the type of database entry to be fetched (this is specified
447       *          on the CLI for the resync command)
448       * @param outputQueue
449       *          a queue of DatabaseChangeRecord objects which will be individually
450       *          fetched via
451       *          {@link #fetchEntry(TransactionContext, SyncOperation)}
452       * @throws SQLException
453       *           if there is an error retrieving the list of entries to resync
454       */
455      @ThreadSafety(level = ThreadSafetyLevel.METHOD_NOT_THREADSAFE)
456      public void listAllEntries(final TransactionContext ctx,
457                                 final String entryType,
458                                 final BlockingQueue<DatabaseChangeRecord>
459                                          outputQueue) throws SQLException
460      {
461        throw new UnsupportedOperationException();
462      }

509       * @param inputLines

514       * @param outputQueue
